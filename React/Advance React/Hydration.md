# 🚀 React Hydration (Interview Notes)

---

# What is Hydration?

Hydration is the **one-time process** where React takes the HTML generated by the server, builds the Fiber tree on the client, attaches event listeners, and connects React to the existing DOM instead of recreating it.

> Hydration makes a server-rendered page interactive.

---

# Why do we need Hydration?

Server sends:

```html
<h1>Ashu</h1>
<button>Like</button>
```

Browser immediately displays:

```text
Ashu

[ Like ]
```

But initially:

❌ No JavaScript

❌ No Event Listeners

❌ Not Interactive

Hydration solves this.

---

# Hydration Flow

```text
User Requests Page

↓

Server Executes React

↓

Generate HTML

↓

Send HTML

↓

Browser Paints HTML

↓

Download JavaScript Bundle

↓

React Executes Components Again

↓

Build Fiber Tree

↓

Compare Fiber Tree ↔ Existing DOM

↓

Match?

↓

YES

↓

Reuse Existing DOM

↓

Attach Event Listeners

↓

Connect React State

↓

Hydration Complete
```

---

# What happens during Hydration?

React performs three major tasks:

## 1. Build the Fiber Tree

React executes the component tree again.

```jsx
<App />
```

↓

Creates a new Fiber Tree.

---

## 2. Attach Event Listeners

```jsx
<button onClick={handleClick}>
```

Now the button becomes interactive.

---

## 3. Connect React with Existing DOM

Instead of:

```text
Delete DOM

↓

Create New DOM
```

React does:

```text
Reuse Existing DOM

↓

Attach React

↓

Interactive
```

---

# Does React create the Fiber Tree from HTML?

❌ No.

React executes your components again on the client.

```text
<App />

↓

React Elements

↓

Fiber Tree
```

Then compares:

```text
Fiber Tree

↔

Existing DOM
```

If they match:

Reuse DOM.

---

# Hydration Mismatch

Server:

```html
<h1>Hello</h1>
```

Client:

```jsx
<h1>Hi</h1>
```

React detects:

```text
Mismatch ❌
```

---

# What happens?

React logs a warning:

```text
Text content did not match.
```

Then React may:

```text
Discard Existing DOM

↓

Client Render

↓

Create New DOM
```

The performance benefit of SSR is lost for that affected subtree.

---

# Common Causes of Hydration Mismatch

## Random Values

```jsx
Math.random()
```

---

## Current Time

```jsx
new Date()
```

---

## Browser APIs

```jsx
window.innerWidth

localStorage
```

Server doesn't have these.

---

## Different Initial State

Server:

```text
Light Theme
```

Client:

```text
Dark Theme
```

Mismatch.

---

# How to Prevent Hydration Mismatch?

Move browser-only logic inside:

```jsx
useEffect(() => {

}, []);
```

Reason:

`useEffect()` only runs on the client after hydration.

---

# Is Hydration a One-Time Process?

✅ Yes.

```text
Initial Page Load

↓

Hydration

↓

Finished
```

After hydration:

```text
setState()

↓

Render

↓

Reconciliation

↓

Commit
```

Normal React lifecycle resumes.

Hydration never runs again for that page load.

---

# What if the DOM doesn't match?

```text
Fiber Tree

↓

Compare Existing DOM

↓

Mismatch

↓

Hydration Warning

↓

Discard Affected DOM

↓

Client Render
```

React cannot safely reuse mismatched DOM.

# ⭐ Interview One-Liners

### Hydration

> Hydration is the process where React connects server-rendered HTML with the client-side React application by building the Fiber tree, attaching event listeners, and reusing the existing DOM.

---

### Hydration Mismatch

> Occurs when the server-rendered HTML doesn't match the client's initial render. React may discard the affected DOM and perform a client-side render instead.

---

### One-Time Process

> Hydration runs only once during the initial page load. All subsequent updates use React's normal render and reconciliation process.
